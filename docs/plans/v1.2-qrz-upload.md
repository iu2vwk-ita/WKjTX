# WKjTX v1.2.0 — qrz.com Logbook upload + eQSL queued retry

**Status:** planning — not implemented.
**Branch:** will spawn `v1.2.0-dev` off `main` when implementation starts.
**Target release:** v1.2.0 (feature release, not a patch).

## 1. Requirements (locked in by user 2026-04-22)

| # | Requirement | Decision |
|---|-------------|----------|
| 1 | qrz.com credentials | Only the **Logbook API Key** (string). No username/password. Generated by the user in qrz.com → My Account → Logbook API → API Key. |
| 2 | UI location | Existing **Impostazioni → Segnalazioni** tab, alongside the existing eQSL block. |
| 3 | Upload mode toggle | Per-service combo: **Automatico su ogni QSO** vs **Solo coda manuale**. Applies to both qrz.com AND eQSL. |
| 4 | Failure handling | No auto-retry. Failed uploads park in a persistent queue; user retries from a dialog. |
| 5 | Scope | Same queue + retry UX covers eQSL (currently auto-only, no retry on network error). |
| 6 | Version | **v1.2.0** — feature release. Not a patch on the 1.1.x line. |
| 7 | App close behaviour | Before exit, if pending queue is non-empty, prompt: **Upload now / Later / Discard**. "Later" keeps queue on disk for next launch. |

Non-goals for v1.2.0:

- Per-profile qrz.com API key — deferred to **v1.3.0** (via `RadioProfileDialog` extension).
- Automatic retry-with-backoff — deferred.
- QKeychain secret storage — deferred. Match existing eQSL pattern (plaintext in `WKjTX.ini`).
- QRZ XML lookup API (callsign info) — out of scope, different API.

## 3. New code modules

### 3.1 `wkjtx/UploadQueue.{hpp,cpp}`

Persistent FIFO of pending uploads. JSON-on-disk.

```cpp
namespace wkjtx {

enum class UploadService { Qrz, Eqsl };

struct QueuedUpload {
    int          id;              // autoincrement, unique
    UploadService service;
    QString      adifRecord;      // full <EOR>-terminated record
    QString      callsign;
    QString      band;
    QString      mode;
    QDateTime    qsoDate;
    int          attempts {0};
    QString      lastError;
    QDateTime    lastAttempt;
};

class UploadQueue : public QObject {
    Q_OBJECT
public:
    explicit UploadQueue (QString const & jsonPath, QObject * parent = nullptr);

    int  enqueue     (QueuedUpload entry);     // returns id
    bool remove      (int id);
    bool markFailed  (int id, QString const & error);
    bool markSuccess (int id);                 // removes

    QVector<QueuedUpload> all () const;
    QVector<QueuedUpload> forService (UploadService s) const;
    int  size () const;

signals:
    void changed ();

private:
    void load ();   // from disk
    void save () const;
    QString  path_;
    QVector<QueuedUpload> entries_;
    int next_id_ {1};
};

} // namespace wkjtx
```

### 3.2 `wkjtx/UploadDispatcher.{hpp,cpp}`

Orchestrator. Knows about Configuration state + holds uploader refs.

```cpp
namespace wkjtx {

enum class UploadMode { Auto, Manual };

class UploadDispatcher : public QObject {
    Q_OBJECT
public:
    UploadDispatcher (Configuration * cfg,
                      UploadQueue * queue,
                      QrzUploader * qrz,
                      EQSL * eqsl,
                      QObject * parent = nullptr);

    // Called from MainWindow::acceptQSO2.
    void onQsoAccepted (QString const & adifRecord,
                        QString const & call,
                        QString const & band,
                        QString const & mode,
                        QDateTime const & qsoDate);

    // Triggered by "Upload pending now" UI or closeEvent "Upload now".
    // Uploads every queued entry. Non-blocking; emits allFlushed() when done.
    void flushPending ();

    // Triggered by "Discard pending" UI. Wipes queue (user confirmed).
    void clearPending ();

    int pendingCount () const;

signals:
    void uploaded     (UploadService s, QString callsign);
    void uploadFailed (UploadService s, QString callsign, QString error);
    void allFlushed   (int ok, int failed);

private:
    void attemptUpload   (QueuedUpload const & q);
    void handleQrzOk     (QString call);
    void handleQrzFail   (QString call, QString err);
    void handleEqslOk    (QString call);
    void handleEqslFail  (QString call, QString err);

    Configuration * cfg_;
    UploadQueue *   queue_;
    QrzUploader *   qrz_;
    EQSL *          eqsl_;
    // In-flight map id → service so signal slots can find the entry.
    QHash<QString, int> inflight_qrz_;
    QHash<QString, int> inflight_eqsl_;
};

} // namespace wkjtx
```

                                 call,

                                 mode,
                                 QSO_date_on);

(The dispatcher's onQsoAccepted replaces BOTH the old eQSL inline
upload and the missing qrz upload — single integration point.)

Hook into `closeEvent`:

```cpp
void MainWindow::closeEvent (QCloseEvent * e)
{
    int pending = m_uploadDispatch->pendingCount ();
    if (pending > 0) {
        int choice = JTDXMessageBox::question_message (
            this, "",
            tr ("%n QSO in coda di upload (qrz.com / eQSL).", "", pending),
            tr ("Cosa fare prima di uscire?"),
            tr ("Upload ora"),
            tr ("Rimanda al prossimo avvio"),
            tr ("Scarta la coda"));
        if (choice == 0) {
            // Spin an event loop until allFlushed() fires or 30 s timeout
            QEventLoop loop;
            QTimer::singleShot (30000, &loop, &QEventLoop::quit);
            connect (m_uploadDispatch, &UploadDispatcher::allFlushed,
                     &loop, &QEventLoop::quit);
            m_uploadDispatch->flushPending ();
            loop.exec ();
        } else if (choice == 2) {
            m_uploadDispatch->clearPending ();
        }
        // choice == 1: do nothing; queue persists on disk for next run
    }
    QMainWindow::closeEvent (e);
}
```

## 6. QRZ API response handling

qrz.com response shapes (from
https://www.qrz.com/docs/logbook/QRZLogbookAPI.html):

**Success:**

```
RESULT=OK&LOGID=12345&COUNT=1&LOGIDS=12345
```

**Failure (known reasons):**

```
RESULT=FAIL&REASON=Invalid+API+key
RESULT=FAIL&REASON=Missing+ADIF+fields
RESULT=REPLACE&LOGID=12345             # existing QSO overwritten
RESULT=AUTH&REASON=invalid+credentials
```

`QrzUploader::uploadAdif` already parses RESULT + REASON and emits the
right signal. `REPLACE` today is treated as failure; should be treated
as **success** (the QSO did get into the logbook — the user may have
already logged the same QSO from another logger). Patch: treat
`RESULT=REPLACE` as `uploaded` in `QrzUploader.cpp:103`.

## 7. eQSL response handling

eQSL endpoint: `https://www.eqsl.cc/qslcard/ImportADIF.cfm`. Returns HTML.

Success indicators (existing JTDX code greps the HTML body):

- "Result: Success" → OK
- "Result: Rejected" or "Error:" → fail

Need to emit the two new signals from `EQSL::networkReply` based on
these patterns. Confirm patterns by inspecting current JTDX code in
`eqsl.cpp` during implementation; the live success/fail detection
already exists as `if/else` branches — just add `emit`.

## 8. Testing plan

Manual tests before shipping v1.2.0:

1. **QRZ happy path** — log FT8 QSO with a valid test API key; verify
   status bar "QRZ: uploaded IU2TEST" + qrz.com Logbook shows the QSO.
2. **QRZ failure** — log with deliberately-wrong API key; verify
   status bar amber toast + queue dialog shows the entry + last_error
   reads "qrz.com: Invalid API key".
3. **Manual mode** — set qrz mode=manual, log QSO, verify NO upload,
   entry lives in queue, retry from dialog succeeds.
4. **Offline** — disconnect network, log QSO auto-mode, verify
   fail-on-enqueue, reconnect, *Upload pending* completes.
5. **Close-with-pending** — leave 3 QSOs in queue, File → Exit:
   verify prompt appears; test all 3 buttons (Upload now / Later /
   Discard); relaunch and confirm queue state matches choice.
6. **eQSL parity** — same 5 tests for eQSL.
7. **Persistence** — crash-kill the app with 2 QSOs in queue
   (Task Manager End Task); relaunch; queue file intact, entries
   present.
8. **JSON corruption resilience** — hand-edit `upload_queue.json`
   to be invalid JSON; relaunch; verify app doesn't crash (fallback
   to empty queue + log warning).

## 9. Risk / unknowns

- **EQSL signal refactor** touches JTDX-inherited code — ensure we
  don't break the existing inline path used by `acceptQSO2`. Mitigation:
  refactor MUST preserve the current behaviour when `m_uploadDispatch`
  is null (i.e., dispatcher is optional; existing direct eqsl->upload
  still works). Dispatcher replaces the call in `acceptQSO2`
  atomically.
- **closeEvent blocking** — a 30 s spin in closeEvent may irritate
  users if the server hangs. Hard cap at 30 s with a visible
  "Uploading... (5s)" progress label. If timeout hits, the remaining
  entries stay queued — NOT discarded.
- **QRZ rate limit** — qrz.com limits are undocumented publicly. A
  30 s flush of 50 entries may trip rate limiting. Add 200 ms delay
  between sequential uploads in `flushPending()`.
- **QSettings password visibility** — API key stored plaintext. Add
  a UI note: "stored in `%LOCALAPPDATA%/WKjTX/WKjTX.ini` — do not
  commit this file to source control". Consider a follow-up issue
  "move to QKeychain / DPAPI" for v1.3.
